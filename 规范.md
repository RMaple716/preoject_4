
# 小组协作编程规范

> 本规范旨在统一团队的 Git 工作流、代码风格及协作流程，减少冲突，提高开发效率与代码可维护性。所有成员请严格遵守。

---

## 一、分支管理规范

我们采用 **GitHub Flow** 简化版工作流，核心分支为 `main`，功能开发基于特性分支进行。

### 1.1 分支类型与命名

| 分支类型 | 命名规则 | 示例 | 说明 |
| :--- | :--- | :--- | :--- |
| **主分支** | `main` | `main` | 保护分支，始终保持可发布状态，仅通过 PR 合并 |
| **功能分支** | `feature/<功能简述>` | `feature/user-login` | 用于开发新功能 |
| **修复分支** | `fix/<问题简述>` | `fix/header-style-bug` | 用于修复非紧急 Bug |
| **热修复分支** | `hotfix/<问题简述>` | `hotfix/payment-crash` | 用于紧急修复线上问题，从 `main` 拉出 |
| **发布分支** | `release/<版本号>` | `release/v1.2.0` | 用于准备发布版本（可选，小团队可省略） |

### 1.2 分支生命周期

1. **新建分支**：从最新的 `main` 分支切出。
   ```bash
   git checkout main
   git pull origin main
   git checkout -b feature/your-feature-name
   ```
2. **保持同步**：定期将 `main` 的更新合并到自己的特性分支，避免偏离过远。
   ```bash
   git checkout main
   git pull origin main
   git checkout feature/your-feature-name
   git merge main
   ```
3. **删除分支**：功能合并入 `main` 后，及时删除远程和本地的特性分支。
   ```bash
   git branch -d feature/your-feature-name
   git push origin --delete feature/your-feature-name
   ```

## 二、提交信息规范

统一提交信息格式，便于生成变更日志和追溯问题。我们遵循 **Conventional Commits** 约定。

### 2.1 提交信息格式

```
<类型>(<范围>): <简短描述>

[详细描述]

[关联 Issue]
```

### 2.2 类型说明

| 类型 | 说明 | 示例 |
| :--- | :--- | :--- |
| `feat` | 新增功能 | `feat(auth): add JWT token refresh logic` |
| `fix` | 修复 Bug | `fix(ui): correct button alignment on mobile` |
| `docs` | 文档更新 | `docs(readme): update installation guide` |
| `style` | 代码格式调整（不影响逻辑） | `style: format with prettier` |
| `refactor` | 重构（不新增功能/修复Bug） | `refactor(api): simplify error handling` |
| `perf` | 性能优化 | `perf(db): add index on user_id column` |
| `test` | 测试相关 | `test(login): add unit tests for validation` |
| `chore` | 构建/工具配置变更 | `chore(deps): upgrade vite to v5` |

### 2.3 提交粒度要求

- **一次提交只做一件事**：避免将多个不相关的修改放在同一个提交中。
- **提交前自查**：
  - 代码是否通过本地编译？
  - 是否已删除多余的 `console.log` 或调试代码？
  - 是否已处理合并冲突？

## 三、代码审查（Code Review）规范

所有合并到 `main` 分支的代码**必须**通过 Pull Request（PR）并至少获得一位成员批准。

### 3.1 PR 发起流程

1. 将本地分支推送至远程仓库：
   ```bash
   git push origin feature/your-feature-name
   ```
2. 在 GitHub/GitLab 上创建 Pull Request，填写以下内容：
   - **标题**：简明扼要说明本次变更（建议与提交类型对齐）。
   - **描述**：说明为什么要做这个改动、做了哪些改动、如何测试。
   - **关联 Issue**：若有关联 Issue，使用 `Closes #123` 语法自动关闭。

### 3.2 审查者职责

- **功能性检查**：逻辑是否正确？边界情况是否考虑？
- **可读性检查**：变量命名是否清晰？函数是否过长？
- **规范检查**：是否符合分支与提交命名规范？
- **测试验证**：必要时拉取代码本地验证。

### 3.3 PR 合并策略

- 优先使用 **Squash and Merge**（将多个提交压缩为一个整洁的提交）。
- 若特性分支提交记录已非常清晰，可使用 **Rebase and Merge** 保持线性历史。
- **禁止**使用普通的 `Merge` 产生无意义的合并提交（`Merge branch 'xxx'`）。

## 四、Issue 管理规范

使用 Issue 跟踪所有任务、Bug 和需求讨论。

### 4.1 Issue 标题格式

- **功能需求**：`[Feature] 功能简述`
- **Bug 报告**：`[Bug] 问题简述 (环境/版本)`
- **优化建议**：`[Enhancement] 优化方向`

### 4.2 Bug 报告需包含内容

- **环境信息**：浏览器版本、操作系统、部署环境（dev/prod）。
- **复现步骤**：1. 打开页面；2. 点击按钮；3. 观察错误。
- **期望行为**：应该发生什么。
- **实际行为**：实际发生了什么（附截图或控制台错误）。

## 五、代码风格统一

为避免因格式化差异造成无意义的代码冲突，小组统一使用自动化工具。

### 5.1 推荐配置（前端示例）

在项目根目录配置 `.editorconfig`、`Prettier` 和 `ESLint`。

```json
// .prettierrc 示例
{
  "semi": true,
  "singleQuote": true,
  "tabWidth": 2,
  "trailingComma": "es5"
}
```

### 5.2 提交前自动格式化

建议配置 `husky` + `lint-staged`，确保提交前代码自动格式化。

```json
// package.json 简单配置
"lint-staged": {
  "*.{js,jsx,ts,tsx}": ["eslint --fix", "prettier --write"]
}
```

## 六、常见协作场景操作指南

| 场景 | 推荐操作流程 |
| :--- | :--- |
| **我要开始一个新功能** | ① 切到 `main` → ② `git pull` → ③ `git checkout -b feature/xxx` |
| **功能开发一半，主分支有更新** | ① `git stash` 暂存当前修改 → ② 拉取并合并 `main` 更新 → ③ `git stash pop` 恢复 |
| **我不小心把代码提交到了错误的分支** | ① `git log` 找到需要移动的提交 ID → ② 切换到正确分支 → ③ `git cherry-pick <commit-id>` → ④ 回到错误分支 `git reset --hard HEAD~1` 删除错误提交 |
| **需要紧急修复线上 Bug** | ① 从 `main` 切出 `hotfix/xxx` → ② 修复并提交 → ③ 发起 PR 合入 `main` → ④ 打 Tag 标记版本 |

## 七、版本发布与 Tag 规范

当 `main` 分支达到可发布状态时，使用 Git Tag 标记版本。

- **版本号规则**：遵循 `v<主版本>.<次版本>.<修订号>`（如 `v1.3.0`）。
- **打标签命令**：
  ```bash
  git tag -a v1.3.0 -m "Release version 1.3.0: add dark mode support"
  git push origin v1.3.0
  ```
